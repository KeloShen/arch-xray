# XRay Rebuilder - 手搓复现指南写作框架

当用户想复现、重做、临摹当前系统时，不要只给一堆文件列表或技术名词。目标是输出一份让人可以照着做的路线图，回答两件事：

1. **应该先做什么**
2. **为什么这样排顺序**

## 文档目标

生成 `xray/docs/guides/rebuild-system.md`，帮助用户从 0 到 1 复现当前系统的核心能力。

文档默认面向：

- 看得懂代码但不熟悉当前项目的人
- 想通过“自己做一遍”理解系统设计的人
- 想做一个简化版 xray / codebase analysis tool 的人

## 核心原则

### 1. 从价值倒推结构

先回答这个系统给用户的最终结果是什么，再解释需要哪些模块来支撑这个结果。

错误写法：

- 先列脚本
- 再列目录
- 最后才说系统在解决什么问题

正确写法：

1. 用户输入一个代码库分析需求
2. 系统扫描仓库，抽取结构、组件和关键关系
3. 系统生成图表和说明文档
4. 用户在 `xray/` 目录获得可阅读成果

### 2. 从 MVP 倒推优先级

把系统拆成：

- **第一版必须有**
- **第二版增强体验**
- **以后再做也不迟**

如果某能力不是复现核心链路必须项，明确写成“暂缓”。

### 3. 从宏观到微观

文档顺序必须遵守：

1. 系统目标
2. 最小产物
3. 模块拆分
4. 模块协作
5. 实现顺序
6. 验证方式
7. 增强路线

不要一开始就进入函数级细节。

## 推荐文档结构

```markdown
# 手搓复现指南：<system-name>

## 1. 先理解你要复现什么
- 这个系统的核心目标
- 用户输入是什么
- 最终输出是什么
- 最小闭环是什么

## 2. 第一版只做什么
- 必做能力
- 可以不做的能力
- 为什么这样裁剪

## 3. 系统拆成哪几个模块
- 模块名
- 责任
- 输入/输出
- 为什么需要这个模块

## 4. 这些模块如何串起来
- 用户请求如何进入系统
- 仓库结构如何被采集
- 图表和文档如何生成
- 输出目录如何组织

## 5. 推荐实现顺序
### Step 1
- What
- Why
- How
- Done

### Step 2
- What
- Why
- How
- Done

## 6. 每一步如何验证
- 命令
- 预期输出
- 失败时先检查什么

## 7. 做完 MVP 之后再加什么
- 架构图增强
- 更多教程文档
- 更深入的代码审查
```

## 对 arch-xray 的建议拆解

写当前仓库时，优先引导用户按这个顺序复现：

### 阶段 1：定义输出契约

先让用户明确 `xray/` 目录长什么样。

原因：

- 没有固定输出，后面的分析和渲染都没有落点
- 输出契约一旦稳定，后面的脚本和文档都更容易围绕它实现

至少要先约定：

- `xray/docs/`
- `xray/assets/diagrams/`
- 几份核心 Markdown
- 至少一张 Context 或 Container 图

### 阶段 2：实现仓库结构采集

把项目树、核心语言、主要模块、入口点提取出来。

当前仓库已有可参考脚本：

- `scripts/collect-structure.sh`

原因：

- 没有结构数据，后面的架构分析只能靠猜
- 这是整个系统后续分析的原材料

### 阶段 3：实现图表生成

先支持最有价值的两张图：

- Context
- Container

当前仓库已有可参考脚本：

- `scripts/generate-c4-diagram.sh`
- `scripts/generate-c4-diagram.ps1`

原因：

- 这两张图最能快速建立系统全貌
- Component/Class 图可以后续再补

### 阶段 4：实现解释型文档生成

至少先产出：

- `architecture-overview.md`
- `project-structure.md`

原因：

- 只有图不够，用户需要文字帮助理解图背后的含义
- 这两份文档最能把“看图”变成“理解系统”

### 阶段 5：实现复现与开发引导

在系统已经能分析和输出成果之后，再补：

- `guides/feature-{name}.md`
- `guides/rebuild-system.md`

原因：

- 没有真实分析产出之前，开发引导很容易变成空泛建议
- 复现指南需要建立在你已经真正理解当前系统之上

## 写作要求

### 每一步必须写清楚四件事

- **What**: 这一阶段交付什么
- **Why**: 为什么现在做
- **How**: 推荐改哪些文件、增加哪些脚本或文档
- **Done**: 用什么现象判断完成

### 必须给出取舍

示例：

- 第一版先不要做 Sequence Diagram，因为 Context + Container 已经足够建立整体理解
- 第一版先不要做多语言教程，只挑项目主语言生成基础教程
- 第一版先不要追求全自动准确架构识别，先接受“半自动 + 人工校正”

### 必须引用当前仓库中的真实对象

优先引用：

- `SKILL.md`
- `scripts/collect-structure.sh`
- `scripts/generate-c4-diagram.sh`
- `references/*.md`
- `examples/`
- `OUTPUT_STRUCTURE.md`

不要写成脱离仓库的通用教程。

## 风格要求

- 用直接语气
- 少讲抽象方法论，多讲具体动作
- 能给顺序就不要只给分类
- 能给判断标准就不要只给建议
- 允许告诉用户“这一步先不要做”
