# 手搓复现 arch-xray 指南

## 1. 先理解你要复现什么

arch-xray 的核心不是“画图”，而是把一个陌生代码库拆成用户能理解、能跟着继续开发的成果包。最小闭环是：

1. 读一个项目
2. 提取结构和关键组件
3. 生成几份说明文档和至少一张可读图表
4. 把结果稳定输出到 `xray/` 目录

如果你一开始就追求所有图、所有教程、所有审查能力，项目会立刻失控。第一版只需要先跑通这个闭环。

## 2. 第一版只做什么

### 必做

- 固定 `xray/` 输出目录
- 采集仓库结构
- 生成 `architecture-overview.md`
- 生成 `project-structure.md`
- 生成一张 Context 或 Container 图

### 先不要做

- 全量 Component/Class 图
- 多语言教程全集
- 细粒度安全审计
- 很复杂的自动推理

原因很简单：这些都建立在前面的结构化分析已经稳定可用之上。

## 3. 系统拆成哪几个模块

### 模块 1：输出契约

- **做什么**：定义 `xray/docs` 和 `xray/assets/diagrams` 的最终形态
- **为什么**：没有稳定输出，前面的分析和后面的渲染都没有落点
- **看哪里**：`OUTPUT_STRUCTURE.md`

### 模块 2：结构采集

- **做什么**：扫描目录、识别语言、入口文件、核心模块
- **为什么**：后面所有分析都依赖这份原始结构数据
- **看哪里**：`scripts/collect-structure.sh`

### 模块 3：图表生成

- **做什么**：把系统边界和容器关系落成 PlantUML 与 SVG
- **为什么**：图是建立系统全貌最快的方式
- **看哪里**：`scripts/generate-c4-diagram.sh`

### 模块 4：解释型文档

- **做什么**：把结构和图表翻译成“人能看懂”的说明
- **为什么**：图只能展示关系，不能自动解释设计取舍
- **看哪里**：`references/architecture-analyst.md`、`references/code-educator.md`

### 模块 5：引导型文档

- **做什么**：告诉用户如何继续开发，或者如何复现这个系统
- **为什么**：这是从“分析”走向“可执行理解”的关键一步
- **看哪里**：`references/dev-guide.md`、`references/xray-rebuilder.md`

## 4. 这些模块如何串起来

用户先提出分析需求，skill 根据需求确定范围和深度。然后系统先跑结构采集，得到目录和模块信息；再根据这些信息生成 Context/Container 图；最后把分析结果整理成 Markdown，统一写到 `xray/docs/`。如果用户想继续开发或自己复现，再额外生成 `xray/docs/guides/` 下的引导文档。

换句话说，链路是：

`需求 -> 结构采集 -> 架构抽象 -> 图表生成 -> 说明文档 -> 开发/复现引导`

## 5. 推荐实现顺序

### Step 1：先固定输出目录和文件名

- **What**：定义 `xray/` 目录结构与核心文件名
- **Why**：这是所有后续脚本和文档的公共契约
- **How**：先写 `OUTPUT_STRUCTURE.md` 风格的结构说明，确保你自己也知道最终要产出什么
- **Done**：你能清楚列出第一版至少会生成哪几份 Markdown 和哪一张图

### Step 2：做结构采集脚本

- **What**：实现“读项目结构”的能力
- **Why**：没有原始结构数据，后面只能凭空猜架构
- **How**：参考 `scripts/collect-structure.sh`，先支持目录树、语言识别、入口点定位
- **Done**：你能对任意仓库输出一份可读的结构摘要

### Step 3：先做最小图表能力

- **What**：生成 Context 或 Container 图
- **Why**：这两张图最适合建立全局认识
- **How**：参考 `scripts/generate-c4-diagram.sh`，坚持使用标准 PlantUML 语法
- **Done**：`xray/assets/diagrams/` 下出现可正常渲染的 `.puml` 和 `.svg`

### Step 4：补两份最重要的解释文档

- **What**：输出 `architecture-overview.md` 与 `project-structure.md`
- **Why**：先把“系统是什么”和“代码在哪里”解释清楚，用户就能开始自主探索
- **How**：分别参考 `references/architecture-analyst.md` 与 `references/code-educator.md`
- **Done**：用户不看源码，也能说出系统核心模块和目录职责

### Step 5：最后再补复现引导

- **What**：输出 `xray/docs/guides/rebuild-system.md`
- **Why**：只有你已经真正理解系统，才能指导别人自己重做一遍
- **How**：按 `What / Why / How / Done` 模板，结合当前仓库真实文件讲实现顺序与取舍
- **Done**：一个新开发者看完文档后，知道第一版该做哪些模块、先后顺序是什么、每一步如何验收

## 6. 每一步如何验证

### 验证输出契约

- 检查 `xray/` 目录结构是否固定
- 检查文档中的图片引用是否为相对路径

### 验证结构采集

- 对当前仓库运行结构采集
- 确认结果里能看到 `scripts/`、`references/`、`examples/` 等关键目录

### 验证图表生成

- 打开生成的 SVG
- 如果看到黑底源码，说明 PlantUML 语法有问题

### 验证解释文档

- 让一个不熟悉项目的人只读文档
- 看他能否回答“系统做什么、核心模块有哪些、应该先看哪里”

### 验证复现指南

- 看文档是否明确区分了“第一版必须做”和“以后再做”
- 看每一步是否都解释了为什么不是别的顺序

## 7. 做完 MVP 之后再加什么

- 增加更多 Component / Class 图
- 增加更细的函数调用链文档
- 增加安全审计和改进建议
- 增加多语言教程
- 增加更强的自动识别和模板化输出

先把最小闭环做实，再谈完整体系。这是复现 arch-xray 最重要的取舍。
